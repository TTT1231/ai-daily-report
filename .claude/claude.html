方式二：Agent 自动化 准备环境与 Key，bun run all 自动抓取、配音、生成图标
安装 Bun + GoTTS 脚本用 Bun，rss/ 采集器用 Go，bun --version 与 go version 都要正常。

📡 抓取来源： rss 按 rss/sources.jsonc 从启用的 RSS 2.0 来源抓取 最近 24 小时 内容；新增 LinuxDo 栏目只需复制来源对象并修改 ID、名称和 URL 经 AI 评分筛选与聚类，最多保留约 15 个 Story（视频主题） 每次运行会基于 rss/rss-state.json 自动去重；LinuxDo 同一 Topic 跨栏目只处理一次，单个来源失败时会保留它的上一次运行状态
网络代理（可选）： 网络受限时，在项目根目录 .env 配置小写 all_proxy=http://127.0.0.1:7890；未配置时 RSS 和 AI 模型请求直接连接 一旦配置 all_proxy，相关请求必须使用该代理，代理无效或不可用时会报错，不会静默回退直连 不会读取 HTTP_PROXY、HTTPS_PROXY、ALL_PROXY 等其他代理变量，也不会自动探测本地代理端口
🎯 兴趣偏好： rss/preferences.jsonc 集中维护重点实体、兴趣信号、不喜欢的内容和评分阈值，文件内包含字段注释 AI 评分 Prompt 与本地关键词保底规则共享同一份兴趣画像，修改偏好无需同步修改 Go 关键词或评分 Prompt
👁️ 图片识别（按需）： 当某个 Story 评分 ≥9、正文较短且含远程图片时，rss 会调用 claude --dangerously-skip-permissions -p --effort low 识别图片内容（识别图片无需扩展思考，降低思考强度以加速），补充事实后再进入下一步解析 ⚠ 因此 Claude CLI 的模型必须支持多模态，并配置图像分析 MCP，否则图片识别步骤会失败（可接受，会降级为纯文本继续，但拿不到图片里的信息）

再看细节

bun run tts 内部做了什么？

generate-tts.mjs 安排顺序，scripts/lib 中的模块各做一件事。

1
读取并校验

paths.mjs 读取新旧数据，report-validation.mjs 检查数据契约。

2
构建 Generated 报表

report-builder.mjs 创建主题、Intro、Outro，并恢复已有图标。

3
逐个处理 Scene

根据字幕和 TTS 参数计算哈希，决定复用旧音频还是调用 MiniMax。

4
计算时间线

按播放顺序写入 audioSrc、timing 和 tts 元数据。

5
事务提交

generated-output.mjs 暂存完整结果，确认成功后才替换正式文件。

Scene 音频怎么决定？
哈希和旧音频都没变？
是：复用不请求 MiniMax，直接使用旧音频时长
否：重新生成minimax-tts.mjs 返回 MP3 与 audioLengthMs
为什么不会写坏旧数据？
全部成功一起提交 audio/ 和 data-generate.json
失败或异常中断恢复旧音频和旧 Generated 数据，避免新旧混合

人工维护 JSONC 采集器自动读取
01
添加自己的 LinuxDo RSS

编辑 rss/sources.jsonc

复制现有 LinuxDo 来源对象，给它一个不重复的 id， 再修改名称和 RSS 地址。同一 LinuxDo Topic 出现在多个栏目时会自动去重。 

{
  "id": "linuxdo-my-category",
  "name": "Linux.do 我的关注栏目",
  "adapter": "linuxdo",
  "url": "https://linux.do/c/example/123.rss",
  "enabled": true,
  "maxPages": 2,
  "pageStart": 0,
  "pageDelaySeconds": 10
}
02
添加重点关注品牌或产品

编辑 priorityEntities

name 用于告诉 AI 这是谁；aliases 是标题中可能出现的写法。标题命中实体后会通过本地兴趣门槛， 再命中重点变化词时会得到 9 分保底。 

{
  "name": "Google Gemini",
  "aliases": ["google gemini", "gemini", "谷歌双子座"]
}
03
添加加分关键词

编辑 signals

任意标题命中后都要直接加分的词，放进 priorityKeywords， 会获得 9 分本地保底。importantChanges 用于重点实体的发布、 开源、价格等变化词；majorPolicy 和 riskEvents 分别用于国家级 AI 政策与风险情报。 

"priorityKeywords": [
  "mcp",
  "你自己的直接加分词"
]
04
添加不喜欢的内容

编辑 dislikes 与 guidance.notInterested

任意标题命中后都不要的词放进 dislikes.hardExclude。 营销和无产品影响的地缘内容可分别维护在 marketing 与 geopolitics；难以用单个词描述的偏好写进 guidance.notInterested，交给 AI 理解。 

"hardExclude": [
  "招聘",
  "你明确不想看的词"
]
05
调整筛选宽松程度

编辑 thresholds

minimumScore 越高，日报越精简；默认 7 分。 maximumCandidates 控制聚类前最多保留多少条候选。 

"thresholds": {
  "minimumScore": 7,
  "maximumCandidates": 30
}
06
让修改生效

运行 RSS 采集器

保存 JSONC 后运行 bun run rss。程序会先校验配置； 字段缺失、空关键词、重复来源 ID、无效 URL 或无效 all_proxy 会直接报错。

bun run rss
运行状态： rss/rss-state.json 不是用户配置，而是采集器自动生成的“上一轮抓取记录”，用于避免每次运行都重复处理最近 24 小时内容 无需维护；删除它会让下一次运行把最近 24 小时内容全部视为新增，并自动生成一份新状态

文件职责

各数据文件的职责划分与维护规则

只人工修改 data-scheme/data.json、data-scheme/images/ 和必要的配置
不要人工修改 data-scheme/data-generate.json
它会在执行 TTS 时被重新生成
不要在 data.json 中维护 audioSrc、timing 或 tts
data-generate.json 是 Remotion 唯一读取的数据文件
data.schema.json 同时描述 Raw 与 Generated 格式
机器字段在 Schema 中可选，但 check-data-json:render 会在渲染前强制要求这些字段存在
rss/sources.jsonc 是人工维护的 RSS 来源入口
新增 LinuxDo 栏目时复制已有来源对象；同一 Topic 会跨栏目自动去重
rss/preferences.jsonc 是人工维护的兴趣画像入口
重点实体、兴趣信号、不喜欢内容和筛选阈值会同时用于 AI 评分与本地保底规则
不要人工修改 rss/rss-state.json
它不是用户配置；删除后下一次运行会自动重建
video-layout.json 是人工维护的项目级固定布局配置
它统一控制视频分辨率与导航容量规则；rss、tts、all、预览和渲染都不会自动修改它。字段含义由 video-layout.schema.json 描述，并由 check-data-json 校验

时间线规则

TTS 时间线计算与场景时长管理

01

TTS 返回的 audio_length

是 Scene 的实际旁白长度

02

Scene 的 durationMs

等于 audioLengthMs + TTS_TAIL_PADDING_MS，默认尾部留白为 250ms

03

startMs 按全部 Story/Scene 的播放顺序累计生成

人不参与维护

04

Header1 和 Header3 的宽度

根据 Generated 数据中的实际 Story 时长自动分配

05

Story 之间的点击音效停留

由 Remotion 组件中的 STORY_TRANSITION_FRAMES 控制，不写入 TTS 时间线，也不写入 Raw 数据
