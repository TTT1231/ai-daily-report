处理流程

跑哪些命令、每个命令做什么

01

bun run video:prepare

一键全流程：归档上期 → 抓取 RSS 生成 data.json → 校验 → 生成旁白 → 生成 Tab 图标。日常出一期日报跑这一条就够，跑完不自动开预览。

02

bun run dev

打开预览。监听 data.json 与图片变化，自动增量生成旁白并刷新画面；想看当前 data-scheme 的效果跑这条。

03

bun run video:render

把当前 data-scheme 渲染导出成 mp4。

04

bun run all:bili

一键发布到 B站：video:prepare → video:render → 生成标题/标签 → 投稿 → 等审核 → 发评论 → 置顶。首次需 biliup 扫码登录。

05

单独跑某一步

bun run rss（只抓取生成 data.json）、bun run tts（只生成旁白和时间线）、bun run generate-svg（只生成 Tab 图标）、bun run check-data-json（只校验）、bun run reset（清空 data-scheme 后完全重建）。

文件职责

各数据文件的职责划分与维护规则

只人工修改 data-scheme/data.json、data-scheme/images/ 和必要的配置
不要人工修改 data-scheme/data-generate.json
它会在执行 TTS 时被重新生成
注意：bun run tts 不只生成音频

命令名容易让人以为它只管"文字转语音"。实际上它是 data-generate.json 的唯一写手，做两件事：

① 装配 generated 数据（report-builder.mjs，不碰音频）：定 theme、构建 intro/outro、从旧数据恢复 icon、按图片真实像素写入 overlayImgWidth/Height；

② 生成音频与时间线（minimax-tts.mjs）：调 MiniMax、写 audioSrc / timing / tts、算 videoStartMs。

两步在同一个原子事务里写入，所以"图片尺寸"虽然不是音频生成的事，却由 tts 触发——换图后重跑 tts（哈希不变走缓存、不花钱）即可重算尺寸。

不要在 data.json 中维护 audioSrc、timing 或 tts
data-generate.json 是 Remotion 的视频数据来源，通过 props 传入
data.schema.json 同时描述 Raw 与 Generated 格式
机器字段在 Schema 中可选，但 check-data-json:render 会在渲染前强制要求这些字段存在
ingest/sources.jsonc 是人工维护的 RSS 来源入口
新增 LinuxDo 栏目时复制已有来源对象；同一 Topic 会跨栏目自动去重；每个来源的 proxy 字段决定抓取时是否走 all_proxy
ingest/preferences.jsonc 是人工维护的兴趣画像入口
重点实体、兴趣信号、不喜欢内容和筛选阈值会同时用于 AI 评分与本地保底规则
不要把 ingest/rss-state.json 当偏好配置修改
它是最近抓取快照；补选可复制条目，偏好请改 preferences.jsonc
video-layout.json 是人工维护的项目级固定布局配置
它统一控制视频分辨率与导航容量规则；rss、tts、all、预览和渲染都不会自动修改它。字段含义由 video-layout.schema.json 描述，并由 check-data-json 校验

时间线规则

TTS 时间线计算与场景时长管理

01

TTS 返回的 audio_length

是 Scene 的实际旁白长度

02

Scene 的 durationMs

等于 audioLengthMs + TTS_TAIL_PADDING_MS，默认尾部留白为 250ms

03

startMs 按全部 Story/Scene 的播放顺序累计生成

人不参与维护

04

Header1 和 Header3 的宽度

根据 Generated 数据中的实际 Story 时长自动分配

05

Story 之间的点击音效停留

由 Remotion 组件中的 STORY_TRANSITION_FRAMES 控制，不写入 TTS 时间线，也不写入 Raw 数据
