数据流

从人工编辑到视频渲染的完整流程

人工编辑
↓
data-scheme/data.json Raw，唯一人工维护源
├─
bun run check-data-json 校验内容字段、Scene/Tab 引用、图片路径；不要求 timing/audioSrc/tts
└─
bun run tts MiniMax TTS + 增量复用
　　├─
读取 Raw data.json
　　├─
读取旧 data-generate.json 作为增量缓存
　　├─
根据 subtitle + TTS 参数计算 Scene 哈希
　　├─
哈希与音频文件均未变化 复用旧 MP3 和 audioLengthMs
　　└─
字幕或 TTS 参数变化 调用 MiniMax 重新生成对应 Scene MP3
　　├─
按 Scene 播放顺序累计 startMs
　　└─
durationMs = audioLengthMs + TTS_TAIL_PADDING_MS 默认尾部留白 250ms
↓
原子写入 data-scheme/data-generate.json
　　├─
audioSrc
　　├─
timing.startMs / timing.durationMs
　　└─
tts 缓存元数据
↓
bun run check-data-json:render 校验 Generated 数据、音频文件和连续时间线
↓
src/daily-report-data.ts 读取 data-generate.json
↓
Remotion Studio / Build / Render

文件职责

各数据文件的职责划分与维护规则

只人工修改 data-scheme/data.json、data-scheme/images/ 和必要的配置
不要人工修改 data-scheme/data-generate.json
它会在执行 TTS 时被重新生成
不要在 data.json 中维护 audioSrc、timing 或 tts
data-generate.json 是 Remotion 唯一读取的数据文件
data-schema.json 同时描述 Raw 与 Generated 格式
机器字段在 Schema 中可选，但 check-data-json:render 会在渲染前强制要求这些字段存在

命令与执行时机

Bun 命令脚本与执行时机

# 编辑 Raw 后，先检查人工内容
bun run check-data-json

# 预览本次 TTS 会 generate 还是 reuse；不调用 API，不改文件
bun run tts:dry-run

# 生成或复用音频，并重新生成 data-generate.json
bun run tts

# 验证 Generated 数据是否可用于渲染
bun run check-data-json:render

# 连续执行 Raw 校验、TTS 和 Generated 校验
bun run prepare-report

# 启动 Studio；predev 会自动执行 Generated 渲染校验
bun run dev

# 构建；prebuild 会自动执行 Generated 渲染校验
bun run build
执行 bun run tts 后，如果 Remotion Studio 已经在运行，需要重启 Studio
否则浏览器可能继续请求旧静态资源版本并出现音频 404 Not Found

时间线规则

TTS 时间线计算与场景时长管理

01

TTS 返回的 audio_length

是 Scene 的实际旁白长度

02

Scene 的 durationMs

等于 audioLengthMs + TTS_TAIL_PADDING_MS，默认尾部留白为 250ms

03

startMs 按全部 Story/Scene 的播放顺序累计生成

人不参与维护

04

Header1 和 Header3 的宽度

根据 Generated 数据中的实际 Story 时长自动分配

05

Story 之间的点击音效停留

由 Remotion 组件中的 STORY_TRANSITION_FRAMES 控制，不写入 TTS 时间线，也不写入 Raw 数据
